# Node-API到ANI迁移背景

## 迁移的背景
### ArkTS语言演进变化
ArkTS-DT是基于TypeScript的编译类型检查规则和JavaScript运行时语义定义的语言。编译前端通过语法糖提供声明式范式扩展，运行时在动态类型基础架构上扩展局部共享内存和并发能力。在ArkTS生态推广发展过程中，开发者对语言的并发能力和性能提出了更高需求。

基于上述的背景，ArkTS语言提供了静态类型的ArkTS-ST版本。

ArkTS-ST主要引入了以下变化：
1. 原生并发共享：通过静态类型实现对象共享和编译优化，提升应用性能。
2. 模块加载优化：支持默认懒加载和多模块并行加载，提升应用启动性能和优化启动内存。
3. 静态类型系统：全面的静态类型检查，减少运行错误，提高编码效率和程序稳定性。
4. 协程语法和API：支持结构化并发，提升并发开发易用性。
5. 增强的语法能力：原生支持声明式UI所需的注解、尾随闭包、具名和匿名扩展函数等。

**ArkTS-DT与ArkTS-ST新老版本CFFI的变化**

ArkTS-ST对静态类型系统和内存共享模型进行了重大改进。继续使用Node-API接口作为CFFI接口不能满足静态版本需求，很多动态特性违反了ArkTS-ST的设计基础。因此，ArkTS-ST版本提供了名为ANI（ArkTS Native Interface）的新接口。

|   | Node-API	| ANI |
|-------|-------|------|
| 接口来源 | 以Node C跨语言接口为基础进行扩展的接口 | 基于类的自研接口，类比于JNI，与语言特性更匹配 |
| 开发范式| 基于动态语言，命令式接口；创建一个对象，无需定义类型 | 基于类声明的开发模式；简洁，命令，也更高效 |
| 强类型 | 接口操作参数就是对象或值；无强制类型要求 | 对类型强校验，与Node-API的映射是1对多；1对1组的关系 |
| 动态特性 | 开发者可以轻松编写动态对象，动态调整对象属性 | 类型不可更改，否则编译无法通过 |
| 基础数据类型 | Node-API的数据类型主要是number类型 | 增加byte、short、int、long、float、double等数据类型 |
| 多线程能力 | 单线程，通过worker、taskpool来支持vm隔离的并行能力 | 并发模型，支持共享对象模型，可以创建多种不同类型的协程任务进行并发执行 |

## 什么场景下需要迁移到ANI
是否需要迁移需要考虑以下使用场景：ArkTS-ST运行模式只能运行基于ANI的模块，ArkTS-DT运行模式下只能运行基于Node-API的模块。如果这个Native模块希望在ArkTS-ST运行模式下运行，需要把Node-API模块迁移到ANI模块；如果希望这个模块在两种模式下都能使用，则需要提供基于两种接口的Native模块。

两种模块可以在开发中共用代码逻辑层。接下来的章节会提供这方面的实践指导。官方也会提供基于IDL的多语言版本绑定开发工具（Taihe工具），以提高迁移效率。
