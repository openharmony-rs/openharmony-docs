## Overview

Starting from API version 26.0.0, HarmonyOS supports intelligent smooth high-speed playback decoding. This capability combines hardware decoding with a human visual perception model to optimize decoding performance at the system level.

In conventional media pipelines, increasing the playback speed proportionally increases the theoretical frame rate that must be processed. For example, playing a 60 fps video at 3.0× requires processing up to 180 frames per second, which introduces the following challenges:

1. On mobile devices, the display frame rate is typically limited by the system's power management policy (commonly 60 fps), which is much lower than the theoretical frame rate required for high-speed playback. During high-speed playback, the human visual system becomes less sensitive to subtle motion because of persistence of vision and reduced motion sensitivity. Decoding and rendering these visually insignificant frames consumes computing resources without noticeably improving the viewing experience.

2. In conventional media pipelines, frame dropping for high-speed playback is typically performed at the rendering stage in the application layer. By that point, the unnecessary frames have already been processed by the decoder. Although applications can perform content-aware frame selection before decoding, doing so generally requires computationally intensive pixel-level analysis, such as optical flow estimation, which increases overall power consumption.

Intelligent smooth high-speed playback decoding eliminates the need for complex pixel-level image analysis. Instead, it leverages motion vectors (MVs) generated by the hardware decoder together with the temporal scalability information in the video bitstream to dynamically adjust the frame retention strategy based on playback speed. This approach reduces decoding and rendering workloads, effectively controls device temperature, and delivers a smoother playback experience than conventional uniform frame dropping.

## Frame Retention Modes

The decoder provides the following frame retention modes. Select the appropriate mode based on your playback scenario.

| Mode | Working Principle | Application Scenario |
| :--- | :--- | :--- |
| **Adaptive (ADAPTIVE)** | Dynamically adjusts the frame retention strategy based on the current playback speed. The system dynamically analyzes video motion characteristics based on the current playback speed passed by the application, and preferentially drops non-key frames with lower visual perception weight. | Recommended for high-speed playback scenarios, providing a smoother experience that better aligns with human visual continuity than uniform frame dropping without content differentiation. |
| **Full (FULL)** | Retains all decoded frames for presentation without applying frame dropping. | Recommended for normal-speed and slow-speed playback to preserve maximum visual quality. Also serves as the default mode during initialization. |
| **Uniform (UNIFORM)** | Uniformly retains frames according to a developer-specified ratio. | Suitable for specialized workload reduction and performance optimization scenarios, such as thermal mitigation under high device temperatures or improving timeline scrubbing smoothness during video editing previews. |

## Parameter Description

This capability takes effect only after you actively deliver the configuration. Configure the following key-value pairs in [OH_AVFormat](../../reference/apis-avcodec-kit/capi-core-oh-avformat.md) to precisely control the behavior.

1. **OH_MD_KEY_VIDEO_DECODER_FRAME_RETENTION_MODE (retention mode)**

   - Type: `int32_t`

   - Value range: See [OH_FrameRetentionMode](../../reference/apis-avcodec-kit/capi-native-avcodec-base-h.md#oh_frameretentionmode).

   - Description: Specifies whether to enable frame retention and its mode. You are advised to configure this parameter during initialization.

2. **OH_MD_KEY_VIDEO_DECODER_SPEED**

   - Type: `double`

   - Value range: Greater than `0.0`.

   - Description: Required only in `ADAPTIVE` mode. Specifies the current playback speed used by the system to dynamically adjust the frame retention strategy.

3. **OH_MD_KEY_VIDEO_DECODER_FRAME_RETENTION_RATIO (retention ratio)**

   - Type: `double`

   - Value range: `[0.01, 1.0]` (for example, `0.5` means retaining 50% of decoded frames).

   - Description: Applies only in `UNIFORM` mode. If the configured value is invalid or out of range, the system defaults to a target frame rate of 30 fps.

## Constraints

- **Supported formats**: This capability is available only for video decoding and currently supports H.264, H.265, and H.266 video streams.

- **Invalid parameter handling**: In `UNIFORM` mode, if the configured frame retention ratio is invalid or not specified, the system defaults to a target output frame rate of 30 fps.

- **Hardware capability fallback**: `ADAPTIVE` mode relies on hardware decoders that provide motion vectors (MVs). On platforms where MV data is unavailable, the system automatically falls back to a basic frame retention strategy while continuing to report `ADAPTIVE` mode externally. In this case, smooth playback is maintained, but content-aware frame retention optimization is unavailable.

- **Audio-video synchronization**: The frame retention mechanism determines only which frames are retained. It preserves the original presentation timestamp (PTS) and decoding timestamp (DTS) of retained frames and therefore does not affect audio-video synchronization.

## High-Speed Playback Development

For typical video playback scenarios, you are advised to adopt a strategy that combines initial configuration with dynamic switching.

- **Initialization**: Before playback starts, configure the decoder to use `FULL` mode to complete the preparation of the underlying feature link.

- **Dynamic switching**: During playback, switch to `ADAPTIVE` mode when the playback speed exceeds 1.0× so that the system can dynamically determine the frame retention strategy. When the playback speed returns to 1.0× or below, switch back to `FULL` mode to retain all decoded frames.

### Initializing the Frame Retention Mode

Before starting playback, configure the initial frame retention mode.

<!-- @[configure_full_baseline](https://gitcode.com/openharmony/applications_app_samples/blob/master/code/BasicFeature/Media/AVCodec/entry/src/main/cpp/capbilities/video_decoder.cpp) -->

``` C++
int32_t VideoDecoder::Configure(const SampleInfo &sampleInfo)
{
    OH_AVFormat *format = OH_AVFormat_Create();
    CHECK_AND_RETURN_RET_LOG(format != nullptr, AVCODEC_SAMPLE_ERR_ERROR, "AVFormat create failed");

    OH_AVFormat_SetIntValue(format, OH_MD_KEY_WIDTH, sampleInfo.videoWidth);
    OH_AVFormat_SetIntValue(format, OH_MD_KEY_HEIGHT, sampleInfo.videoHeight);
    OH_AVFormat_SetDoubleValue(format, OH_MD_KEY_FRAME_RATE, sampleInfo.frameRate);
    OH_AVFormat_SetIntValue(format, OH_MD_KEY_PIXEL_FORMAT, sampleInfo.pixelFormat);
    OH_AVFormat_SetIntValue(format, OH_MD_KEY_ROTATION, sampleInfo.rotation);
    if (sampleInfo.codecSyncMode) {
        OH_AVFormat_SetIntValue(format, OH_MD_KEY_ENABLE_SYNC_MODE, sampleInfo.codecSyncMode);
    }
    if (sampleInfo.isSmartFluencySupported) {
        // Configure FULL mode to prepare the runtime environment for optimal ADAPTIVE performance.
        OH_AVFormat_SetIntValue(format, OH_MD_KEY_VIDEO_DECODER_FRAME_RETENTION_MODE,
                                OH_FRAME_RETENTION_MODE_FULL);
    }

    int ret = OH_VideoDecoder_Configure(decoder_, format);
    OH_AVFormat_Destroy(format);
    format = nullptr;
    CHECK_AND_RETURN_RET_LOG(ret == AV_ERR_OK, AVCODEC_SAMPLE_ERR_ERROR, "Config failed, ret: %{public}d", ret);
    return AVCODEC_SAMPLE_ERR_OK;
}
```

### Switching Frame Retention Modes Dynamically

You can switch frame retention modes while the decoder is running by calling `OH_VideoDecoder_SetParameter`. Parameter updates take effect according to the behavior defined for this API.

<!-- @[onUserSpeedChanged](https://gitcode.com/openharmony/applications_app_samples/blob/master/code/BasicFeature/Media/AVCodec/entry/src/main/cpp/capbilities/video_decoder.cpp) -->

``` C++
int32_t VideoDecoder::OnUserSpeedChanged(double targetSpeed)
{
    OH_AVFormat *param = OH_AVFormat_Create();
    CHECK_AND_RETURN_RET_LOG(param != nullptr, AVCODEC_SAMPLE_ERR_ERROR, "AVFormat create failed");

    // Introduce epsilon to handle precision comparison for the double type.
    const double EPSILON = 1e-6;

    if (targetSpeed > 1.0 + EPSILON) {
        // Scenario: High-speed playback (e.g., 1.5x, 2.0x, 3.0x).
        // Strategy: Enable the adaptive mode.
        OH_AVFormat_SetIntValue(param, OH_MD_KEY_VIDEO_DECODER_FRAME_RETENTION_MODE,
                                OH_FRAME_RETENTION_MODE_ADAPTIVE);
        OH_AVFormat_SetDoubleValue(param, OH_MD_KEY_VIDEO_DECODER_SPEED, targetSpeed);
    } else {
        // Scenario: Normal playback (1.0x) or slow playback (<1.0x).
        // Strategy: Switch to the full mode to ensure every frame is sent for display.
        OH_AVFormat_SetIntValue(param, OH_MD_KEY_VIDEO_DECODER_FRAME_RETENTION_MODE,
                                OH_FRAME_RETENTION_MODE_FULL);
    }

    // Deliver parameters in real time to dynamically adjust the system frame retention strategy.
    int32_t ret = OH_VideoDecoder_SetParameter(decoder_, param);
    OH_AVFormat_Destroy(param);
    CHECK_AND_RETURN_RET_LOG(ret == AV_ERR_OK, AVCODEC_SAMPLE_ERR_ERROR,
                             "SetParameter failed, ret: %{public}d", ret);
    return AVCODEC_SAMPLE_ERR_OK;
}
```

## Additional Use Cases

In addition to high-speed playback, this capability can also improve performance and responsiveness in the following scenarios.

- **Thermal mitigation**: When the device enters a thermal throttling state, reducing power consumption becomes more important than maintaining a high frame rate. In this case, you can switch to `UNIFORM` mode and configure a lower frame retention ratio (for example, `0.3`). By reducing the number of frames decoded and presented, the decoder lowers the overall system workload, helping reduce device temperature.

- **Improving timeline scrubbing responsiveness**: In video editing scenarios where users frequently scrub the playback timeline, decoding every intermediate frame can noticeably reduce UI responsiveness. You can temporarily switch to `UNIFORM` mode and configure a lower frame retention ratio to reduce the number of intermediate frames that are decoded. This allows the preview to update more quickly, resulting in smoother and more responsive timeline scrubbing.

**Example (responding to a thermal warning):**

int32_t VideoDecoder::OnThermalWarningReceived(double ratio)
{
    OH_AVFormat *param = OH_AVFormat_Create();
    CHECK_AND_RETURN_RET_LOG(param != nullptr, AVCODEC_SAMPLE_ERR_ERROR, "AVFormat create failed");

    // Use the UNIFORM mode.
    OH_AVFormat_SetIntValue(param, OH_MD_KEY_VIDEO_DECODER_FRAME_RETENTION_MODE,
                            OH_FRAME_RETENTION_MODE_UNIFORM);

    // Set the retention ratio (for example, 0.3 means only 30% of decoded frames are output).
    OH_AVFormat_SetDoubleValue(param, OH_MD_KEY_VIDEO_DECODER_FRAME_RETENTION_RATIO, ratio);

    // The configuration takes effect in real time, and the system performs uniform frame dropping to reduce the overall system load.
    int32_t ret = OH_VideoDecoder_SetParameter(decoder_, param);
    OH_AVFormat_Destroy(param);
    CHECK_AND_RETURN_RET_LOG(ret == AV_ERR_OK, AVCODEC_SAMPLE_ERR_ERROR,
                             "SetParameter failed, ret: %{public}d", ret);
    return AVCODEC_SAMPLE_ERR_OK;
}
