A key encapsulation mechanism (KEM) is a cryptographic mechanism that securely establishes a shared secret between two parties using asymmetric keys. Unlike traditional key agreement, key encapsulation is a one-way operation: the encapsulating party uses the public key to generate a ciphertext and a shared secret, and the decapsulating party uses the private key to recover the shared secret from the ciphertext.

Differences between key encapsulation and traditional key agreement:

- **Key agreement (for example, ECDH, DH, X25519):** A two-way negotiation where each party holds a private key, exchanges public keys, and then independently computes the same shared secret.

- **Key encapsulation (for example, ML-KEM):** A one-way operation where the encapsulating party uses the peer's public key to generate a ciphertext and a shared secret, and sends the ciphertext to the peer; the decapsulating party uses the private key to recover the shared secret from the ciphertext, without requiring a bidirectional exchange of public keys.

Module-Lattice-Based Key-Encapsulation Mechanism (ML-KEM) is a post-quantum cryptographic algorithm defined by the NIST FIPS 203 standard, designed to replace traditional key agreement algorithms to resist quantum computing attacks.

The shared secret generated by key encapsulation can be handled in two ways:

- **Stored to HUKS:** Specify **sharedKeyAlias** and **sharedKeyParams** during encapsulation or decapsulation. The shared secret is stored in HUKS under the specified alias, and can be directly used for subsequent operations such as encryption and decryption via that alias. In this mode, the **sharedSecret** field in the encapsulation and decapsulation results is empty. For detailed development procedure, refer to the example of hosting the shared secret in HUKS in [Encapsulating a Key (ArkTS)](./huks-kem-arkts.md).

- **Exported to the app:** Do not specify **sharedKeyAlias** and **sharedKeyParams** during encapsulation or decapsulation. The shared secret is exported to the app through **sharedSecret** in the encapsulation and decapsulation results, and the app manages the storage and usage of the shared secret on its own. In this mode, the app must manage the key by itself. For detailed development procedure, refer to the example of not hosting the shared secret in HUKS in [Encapsulating a Key (ArkTS)](./huks-kem-arkts.md).

## Supported Algorithms

The following table lists the specifications supported for key encapsulation.

<!--Del-->

The vendor-adapted key management service specifications for OpenHarmony are divided into mandatory and optional specifications. Mandatory specifications are algorithm specifications that all vendors must support. Optional specifications, on the other hand, are implemented at the vendor's discretion. If you need to use an optional specification, consult the documentation provided by the specific vendor to confirm support before use.

**You are advised to use mandatory specifications when developing apps to ensure full platform compatibility.**
