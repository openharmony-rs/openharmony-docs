When developing the UI using NDK, you can enable the UI to respond to various interaction events, including basic input events, gesture events, and drag events, by adding event listeners and event handling logic to components.

## Adding Event Listeners

NDK APIs provide event listening capabilities for components. You can receive and process events by adding event listeners and registering event types.

### Listening Methods

The NDK APIs support the following two event listener methods.

- **Listening to node events**: Adds an event listener to a single component, listening only to events that occur on that component. Use [addNodeEventReceiver](../reference/apis-arkui/capi-arkui-nativemodule-arkui-nativenodeapi-1.md#addnodeeventreceiver) to add the listener, and then use [registerNodeEvent](../reference/apis-arkui/capi-arkui-nativemodule-arkui-nativenodeapi-1.md#registernodeevent) to declare the event types to listen for.

- **Listening to global events**: Listens for event triggers from all native components. Use [registerNodeEventReceiver](../reference/apis-arkui/capi-arkui-nativemodule-arkui-nativenodeapi-1.md#registernodeeventreceiver) to register a global event handler, enabling unified processing of events from various components.

In addition to component interaction events, the NDK APIs also support listening for component layout completion events, frame display events, and system dark/light mode change events.

## Binding Interaction Events

On top of adding event listeners, you can bind various interaction events to respond to user input behaviors. Interaction events are mainly divided into three categories: basic input events, gesture events, and drag events.

### Basic Input Events

Basic input events are raw events directly generated by users through different peripherals (such as touchscreens, mice, keyboards, etc.). Based on their characteristics, they are generally divided into the following two categories.

- **Pointing events**: Events that contain location information about where the event occurred. The dispatch target is determined by the component hit when the first event of this interaction occurs. These include: touch events, mouse events, and axis events.

- **Non-pointing events**: Events without specific location information. The dispatch target is the currently focused component. These include: key events.

A basic input event typically starts with an event that marks the beginning and ends with an event that marks the conclusion, with movement or update events continuously reported in between. For pointing events, the system ensures as much as possible that the component participating in the response receives a complete sequence. However, for non-pointing events, the start and end may be dispatched to different components due to focus changes.

Basic input events are delivered through the response chain. The event is processed first by the innermost component and then passed layer by layer to the parent component. Any component in the chain can actively terminate the further delivery of the event.

### Gesture Events

A gesture is an interaction result recognized when a series of basic input events, continuously reported and accumulated, meet certain characteristics. For example, a tap: pressing down and lifting up within a relatively short time.

Gestures can mask the differences between various basic input events and provide higher-level interaction capabilities. The NDK APIs support two types: single gestures and combined gestures. Single gestures include tap, swipe, long press, pinch, rotation, and fling. Combined gestures allow multiple gestures of different types to be grouped together, and the relationships between gestures within the group are controlled by setting a recognition mode (sequential, parallel, or exclusive).

Multiple gestures can be bound to a single component. These gestures are collected through hit testing when the user presses down and are managed uniformly by the system. All gestures continuously receive input events until one gesture meets its conditions. After that, only this gesture can continue to receive and process input events. In addition, the NDK APIs also support custom gesture determination, allowing dynamic decision-making on whether a gesture recognizer should be recognized as successful by the system just before it is about to trigger successfully.

### Drag Events

A drag event refers to the interactive behavior where the user moves an object from a source position to a target position. The NDK APIs provide the following two drag methods.

**General drag:** Enables a component to be draggable through the component's drag attribute settings. Corresponding component events are triggered at different stages of the drag (such as pre-drag, drag start, enter, move, leave, drop, and end). It supports custom drag preview images and drag tracking image options, and data is transferred through Unified Data Management Framework (UDMF).

**Actively initiate drag via DragAction:** The ability to actively initiate a drag through an event. By registering the [NODE_ON_TOUCH_INTERCEPT](../reference/apis-arkui/capi-native-node-h.md#arkui_nodeeventtype) event, a **DragAction** object is created after receiving a touch event. Set the drag tracking image options, tracking point, and drag data, then actively initiate the drag. This method provides more flexible drag control, enabling scenarios where drag is triggered on demand.
