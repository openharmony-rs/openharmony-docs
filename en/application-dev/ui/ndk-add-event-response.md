When developing a UI with the NDK, you can add event listeners and event handling logic to components, enabling the UI to respond to various interaction events, including basic input events, gesture events, and drag events.

## Adding an Event Listener

NDK APIs provide the event listening capability for components. You can add event listeners and register event types to receive and process events.

### Listening mode

NDK APIs support the following event listening modes:

- **Node event listening**: Add an event listener for a single component to listen to only the events that occur on the component. Use [addNodeEventReceiver](../reference/apis-arkui/capi-arkui-nativemodule-arkui-nativenodeapi-1.md#addnodeeventreceiver) to add a listener and use [registerNodeEvent](../reference/apis-arkui/capi-arkui-nativemodule-arkui-nativenodeapi-1.md#registernodeevent) to declare the type of event to be listened for.
- **Global event listening**: Listen to event triggers of all native components. Use [registerNodeEventReceiver](../reference/apis-arkui/capi-arkui-nativemodule-arkui-nativenodeapi-1.md#registernodeeventreceiver) to register a global event handler that can uniformly handle events from various components.

In addition to component interaction events, the NDK API also supports listening to component layout completion events, drawing display events, and system light/dark mode change events.

## Binding Interaction Events

In addition to adding event listeners, you can bind various interaction events to respond to user input. Interaction events are classified into basic input events, gesture events, and drag events.

### Basic Input Events

Basic input events are primitive events directly generated by users through different peripherals (such as touchscreen, mouse, and keyboard). Based on their characteristics, they are generally divided into two types:

- **Pointer events**: events that carry location information indicating where the event occurred. The dispatch target is the component hit by the first event in the sequence. Such events include touch events, mouse events, and axis events.
- **Non-pointer events**: events that do not contain specific location information. The dispatch target is the currently focused component. Such events include key events.

A basic event usually starts with an event marked as the start and ends with an event marked as the end, with movement or update events continuously reported in between. For pointer events, the system ensures that components participating in the response receive the complete event sequence. However, for non-pointer events, the start and end may be dispatched to different components due to potential focus changes during the event sequence.

Basic input events are delivered through a response chain. The event is first processed by the innermost component and then passed up to parent components layer by layer. Any component in the chain can actively terminate further event delivery.

### Gesture Events

A gesture is an interaction pattern recognized when a series of basic events meet specific characteristics. For example, a click is detected when a press and release occur within a short timeframe.

Gestures can mask the differences between various basic events and provide more advanced interaction capabilities. NDK APIs support both single gestures and combined gestures. Single gestures include click, swipe, long press, pinch, rotate, and swipe. Combined gestures can combine multiple gestures of different types. The relationship between gestures can be controlled by setting the recognition mode (sequential, parallel, or mutually exclusive) for the gestures in the group.

Multiple gestures can be bound to a single component. When a user presses, these gestures are collected via hit testing and managed uniformly by the system. All gestures continuously receive input events until one gesture meets the recognition condition; after that, only that gesture continues to receive and process input events. In addition, the NDK APIs support custom gesture judgment, allowing you to dynamically decide whether a hand gesture recognizer should be considered successful by the system when it is about to be triggered successfully.

### Drag Events

Drag events refer to the interaction where a user moves an object from a source location to a target location. NDK APIs provide the following drag modes:

**General drag**: You can enable the drag capability for a component by setting its drag attributes. Different stages of a drag trigger corresponding component events (e.g., pre‑drag, drag start, enter, move, leave, drop, drag end). Custom drag preview and drag‑along image options are supported, and data transfer uses the uniform data management framework (UDMF).

**DragAction – active drag initiation**: Actively initiate dragging through an event. Register the [NODE_ON_TOUCH_INTERCEPT](../reference/apis-arkui/capi-native-node-h.md#arkui_nodeeventtype) event, create a **DragAction** object after receiving a touch event, set the drag‑along image, drag pointer location, and drag data, and then actively start dragging. This method provides more flexible drag control, enabling drag‑on‑demand scenarios.
